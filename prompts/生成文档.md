### 文档生成说明

本次文档生成遵循以下规范：

#### 文档结构

1. **doc/summary.md** - 项目概述文档
    - 使用 h3 及以下标题级别
    - 按以下顺序组织内容：项目概述 → 核心架构与流程 → 核心概念详解 → 主要特性 → 使用方法
    - 具体章节根据项目实际情况调整，以上为推荐的基本框架

2. **doc/api.md** - API 文档
    - 使用 h3 及以下标题级别
    - 按以下顺序组织内容：配置项 → 接口/命令 → 程序化 API → 数据模型 → 机制说明
    - 具体章节根据项目实际情况调整，以上为推荐的基本框架

#### 标题与层级规范

- 仅使用 h3（###）及以下标题级别，不使用 h1、h2
- h3 用于大章节，h4 用于小节，h5 用于子项
- 标题应简短精炼，直接表达该节内容

#### 表格规范

- 表格是展示结构化信息的首选方式，优先使用表格而非纯文字列表
- 根据上下文选择合适的表格列组合：

| 场景 | 推荐列组合 |
|------|-----------|
| 配置项/参数 | 属性名、类型、必填、默认值、说明 |
| 数据模型字段 | 属性名、类型、说明 |
| 枚举/映射关系 | 名称/key、值/表达式、说明 |
| 方法列表 | 方法签名、说明 |
| 特性对比 | 特性、说明 |
| 依赖/关联关系 | 实体、来源/目标、说明 |

- 表格中类型使用简洁标注：`string`、`number`、`boolean`、`object`、`function`、`Date`、`array`
- 联合类型用 `/` 分隔，如 `string / object`
- 必填列标注 `是` 或 `否`
- 默认值列用反引号包裹代码值，如 `true`、`1000`
- 说明列应简洁，如需补充较长说明可在表格外用段落或引用块展开

#### 架构与流程图

- 使用 ASCII 字符画展示数据流、依赖关系、层级结构
- 用箭头 `↓` `→` `↙` `↘` 表示流向
- 用括号标注关键触发条件或时机
- 图后可接表格详细说明图中各节点/关系

#### 代码示例规范

- 代码示例需有上下文注释，说明场景和目的
- 展示典型用法，包含必要的配置参数
- 多个相关示例按复杂度递进排列（简单 → 常用 → 高级）
- 输出示例应完整且格式化，反映真实返回结构
- 代码块注明语言标识（`js`、`json` 等）

#### 重要说明与注意事项

- 使用引用块（`>`）标注关键设计决策、常见陷阱、与直觉不同的行为
- 加粗标注核心概念关键词
- 示例格式：`> **关键设计**：用一句话概括核心决策及原因。`

#### API 文档编写规范

- HTTP 接口格式：`方法 路径`，路径中的可变部分用花括号包裹
- 接口说明先写一句话概述功能
- 请求参数用表格列出，包含必填和默认值列
- 返回格式用代码块展示，并在代码前用文字说明不同条件下的格式差异
- 程序化 API 按模块分组，每个方法一行表格说明
- 方法签名需体现参数结构，如 `module.method(param1, opts)`

#### 数据模型规范

- 每个模型独立一个 h4 小节
- 用表格列出字段：属性名、类型、说明
- 在表格下方补充索引、唯一约束等关键信息
- 说明字段中的业务语义，而非仅重复字段名

#### 内容来源与准确性

- 内容必须基于实际代码分析，确保准确性
- 不臆测或虚构不存在的功能、参数、返回值
- 如代码中有默认值，文档中必须与代码一致
